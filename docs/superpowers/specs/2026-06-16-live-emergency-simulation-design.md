# AI Emergency Commander 实时交互仿真设计

## 1. 目标

把现有“预计算完整时间线后查看”的 Streamlit 页面升级为可交互的实时救援仿真。验收时，用户只需点击一次即可生成随机地图并自动运行算法；运行过程中可以随时注入突发事件，系统从当前状态重新推理、规划并继续执行，最后输出可下载的结果报告。

## 2. 验收场景

1. 用户打开页面，点击“随机生成并启动”。
2. 页面立即生成一张新的、连通的随机救援地图，并显示本次随机种子。
3. 仿真无需再次点击开始，按固定时间间隔自动推进。
4. 左侧地图实时显示车辆、无人机、任务路线、风险道路和受灾区域。
5. 右侧算法演示逐阶段展示输入校验、贝叶斯推理、风险排序、风险感知 A*、效用计算、任务分配和任务执行。
6. 用户可随时点击道路坍塌、火势蔓延、新增求救或无人机情报按钮。
7. 默认模式自动选择对当前任务影响最大的有效目标；高级模式允许手动指定道路或区域。
8. 突发事件应用后，受影响任务被中断，算法基于当前单位位置和最新场景重新规划，然后自动继续执行。
9. 所有可完成救援任务结束，或系统确认不存在可行任务时，仿真停止并展示最终报告。

## 3. 方案选择

采用实时增量仿真会话，而不是预计算动画或手动“下一步”。

- 预计算动画无法保证运行中插入事件后，未来状态仍然基于真实的当前状态。
- 手动推进容易测试，但不符合点击一次后自动运行的验收标准。
- 增量会话可在 Streamlit 重运行模型中持久保存场景、时钟、单位状态、算法阶段和事件记录，并通过 `st.fragment(run_every=...)` 自动推进。

## 4. 系统结构

### 4.1 随机场景生成器

新增独立的随机场景生成模块，输入整数随机种子，输出符合现有场景数据契约的字典。

生成规则：

- 生成 3 个受灾区域、1 个指挥中心、1 个医院和必要的中继节点。
- 至少生成 2 辆救援车和 1 架无人机。
- 地面道路先构造连通骨架，再为每个受灾区域增加独立备选入口；任意一条地面道路坍塌后，指挥中心到每个受灾区域仍至少保留一条可达路径。
- 空中航线保证无人机可以到达所有受灾区域。
- 区域观测、道路时间、火灾风险、损毁风险和资源参数在合法范围内随机生成。
- 相同种子必须生成完全相同的场景，便于复现实验和失败案例。

### 4.2 实时仿真会话

新增与 Streamlit 解耦的 `LiveSimulation`。它持有：

- 当前标准化场景；
- 仿真时钟；
- 单位状态；
- 当前规划结果；
- 当前算法阶段；
- 待处理事件；
- 时间线、算法日志和重规划日志；
- 运行、暂停、完成或无法继续等会话状态。

核心公共操作：

- `create(...)`：初始化会话，但不预计算未来事件；
- `step()`：推进一个算法阶段或一个执行时钟片；
- `inject_event(...)`：从当前状态应用一个突发事件并进入重规划；
- `build_result()`：产生最终报告和兼容的决策输出。

### 4.3 算法状态机

每次初始规划或重规划依次经过以下阶段：

1. `validate`：标准化并校验场景输入。
2. `infer`：离散化证据并执行完整贝叶斯网络精确推理。
3. `prioritize`：根据被困概率、生命风险和道路可通概率计算区域优先级。
4. `route`：使用风险感知 A* 生成地面或空中候选路线。
5. `utility`：计算每个单位到每个区域的期望效用及分项贡献。
6. `allocate`：执行全局任务分配并启动任务。
7. `execute`：按固定分钟步长推进车辆和无人机有限状态机。
8. `replan`：突发事件发生后废止受影响任务，并重新进入 `infer`。
9. `complete`：满足终止条件后冻结最终结果。

算法阶段不能只作为装饰标签。每次 `step()` 必须执行该阶段对应的真实计算，并把摘要写入算法日志；右侧演示面板直接读取该日志。

### 4.4 突发事件

支持现有四类事件：

| 按钮 | 自动目标策略 | 场景变化 |
| --- | --- | --- |
| 道路坍塌 | 优先选择当前地面任务正在使用的开放道路，否则选择高流量开放道路 | 道路状态变为阻断 |
| 火势蔓延 | 选择当前优先级最高且未完成的区域 | 提高区域火灾证据或相关风险 |
| 新增求救 | 选择当前被困证据较低且未完成的区域 | 提高 SOS 和被困相关观测 |
| 无人机情报 | 选择当前不确定性最高的未完成区域 | 更新无人机影像与道路观测 |

高级模式显示有效目标下拉框。事件按钮在运行和暂停状态均可使用；完成后不可再注入事件。无有效目标时按钮禁用并说明原因。

道路坍塌若命中活动路线，相关单位保留当前安全节点和可视位置，但当前任务立即失效。其他未受影响单位继续保持任务。系统记录旧方案、事件、失效原因和新方案。

## 5. 页面设计

延续现有工业化离线指挥台风格，重组为三个主要区域：

### 顶部控制区

- 主按钮：“随机生成并启动”；
- 暂停/继续和重新生成；
- 运行速度选择；
- 当前随机种子、仿真时间、会话状态和重规划次数。

点击主按钮后自动开始，不出现第二个必需的“开始”步骤。

### 实时工作区

- 左侧约 62%：地图、移动单位、当前路线和事件影响高亮；
- 右侧约 38%：九阶段算法轨道、当前阶段说明、最新计算摘要和滚动日志；
- 当前阶段使用明确的高亮和进度动画，已完成阶段保留时间戳，尚未执行阶段保持弱化。

### 事件与结果区

- 四个醒目的突发事件按钮；
- “高级目标选择”折叠区；
- 活动任务、候选效用和事件历史；
- 完成后自动切换到结果摘要，并提供 JSON 与 Markdown 下载。

页面必须在常见桌面宽度下无需水平滚动；窄屏时地图和算法面板纵向排列。

## 6. 自动推进与 Streamlit 状态

`st.session_state` 只保存可序列化的仿真会话数据和界面设置。自动运行区域使用 `st.fragment(run_every=...)`：

- 会话为 `running` 时，每次刷新调用一次 `step()`；
- 暂停时保持渲染但不推进；
- 点击事件按钮时，先把事件注入当前会话，再由后续刷新执行重规划；
- 完成后取消自动推进，并显示固定最终结果；
- 页面普通重运行不能重置随机地图或丢失仿真进度。

## 7. 完成条件与结果报告

区域救援完成以救援车完成现场服务并把载员送达医院为准；无人机侦察完成不等于区域救援完成。会话在以下任一条件满足时结束：

- 所有仍需救援的区域均完成救援；
- 没有活动任务，且所有未完成区域对所有可用单位均不可行；
- 达到防止无限运行的最大仿真分钟数或最大步数，并在报告中标记为超时。

最终报告至少包含：

- 场景编号与随机种子；
- 结束原因与总仿真时间；
- 已完成和未完成区域；
- 估计救援人数；
- 每个单位的任务数、最终状态和行驶时间；
- 初始规划与最终规划；
- 所有突发事件及每次重规划前后的路线变化；
- 最终贝叶斯概率、任务效用和不可行原因；
- 完整时间线和算法日志。

## 8. 错误处理

- 随机场景生成后必须经过现有 JSON 契约与标准化校验；失败时使用同一种子报告错误，不静默换场景。
- 单条路线不可行时保留候选的不可行原因，其他候选继续参与分配。
- 事件目标已失效或不存在时拒绝事件，不修改会话，并在页面显示可恢复提示。
- 单个阶段异常时暂停会话、记录阶段和异常信息，允许用户重新生成；不能继续显示为运行中。
- 任何情况下不得因为 Streamlit 重运行而重复应用同一个事件。

## 9. 测试与验收证据

### 单元测试

- 相同种子生成相同场景，且不同种子产生不同地图。
- 随机场景通过数据契约，并保证地面骨架与空中航线可达。
- 会话严格按状态机推进，初始创建不会预计算未来。
- 执行阶段会真实改变仿真时间和单位位置。
- 四类事件都能从当前状态应用并触发重规划。
- 道路坍塌只废止受影响的活动任务。
- 暂停、完成和超时状态不会继续推进。
- 最终报告区分侦察完成与救援完成。

### 集成测试

- Streamlit 应用能够启动且没有运行时异常。
- 一次主按钮点击同时生成地图并进入运行状态。
- 自动刷新后算法阶段、时钟和单位位置发生变化。
- 运行中点击事件后，事件计数和重规划日志增加，仿真随后继续。
- 完成后显示结果摘要和下载控件。

### 浏览器验收

使用真实浏览器执行完整流程并保存截图：

1. 初始空闲页；
2. 点击一次后的随机地图和正在运行的算法阶段；
3. 单位已移动的中间状态；
4. 注入道路坍塌后的重规划状态；
5. 最终结果报告及下载按钮。

只有自动化测试和上述浏览器流程都通过，才能认定网页达到验收标准。

## 10. 非目标

- 不接入在线地图、实时灾害 API 或云端服务；
- 不实现多人协作或持久化数据库；
- 不把事件改造成预先排定的脚本；
- 不更换现有贝叶斯网络、A* 和期望效用算法，只调整其调用方式以支持增量运行。
